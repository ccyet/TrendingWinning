TrendingWinning 使用指南

TDX-only A 股 K 线趋势策略工作台，覆盖数据落地、审计补齐、策略扫描、单策略回测、组合策略回测、参数遍历和 Docker 部署。

当前设计原则

路径选择使用文件夹选择框；同一任务的周期、代码、日期放在同一行；趋势、通道、区间、反转四类形态识别彼此解耦。

能力总览 本地运行 路径选择 TDX K线 策略扫描 单策略回测 组合策略回测 回测K线指南 输出产物 Docker Win/TDX 注意

能力总览

TDX 数据日 K 1d 和分钟 K 5m / 15m / 30m / 60m，不使用 akshare 通道；高分钟周期可由 TDX 5m 聚合。
数据管理本地缓存库存、parquet 审计、按需补齐、K 线覆盖率检查、最长连续缺口定位、日 K 锚定分钟交易日。
过滤规则按主板、科创、创业、北交所涨停开盘规则过滤污染交易日。
独立形态识别趋势、区间、通道、反转模块只输出标准事件，策略层统一消费。
末端假突破过滤可选开仓过滤。趋势或通道已经走得较久，价格贴近边缘、远离中轴、突破推进不足并带明显上影线/下影线时，拒绝本次开仓。
挂单触发突破在信号 K 上方/下方挂单，撮合层按下一根 K 是否触发成交。
策略执行空间单策略和组合策略运行前会把当前样本、识别形态、信号条件、触发成交、开仓过滤、退出条件、仓位规则、或然分支和复盘输出整理成一张表，先确认哪些模块会生效。
回测结果先看策略K线运行区间，再看核心绩效概览、净值曲线、回撤曲线、回撤区间明细、实验诊断摘要、交易路径分布、数据覆盖率概览、订单决策概览、拒绝原因分布、逐笔交易、策略绩效、识别模块绩效、信号形态绩效、订单决策统计和策略过滤统计；K 线支持横轴/价格轴缩放、SVG 矢量高清展示，并标注开多、开空、止损和盈利通道回撤止盈。回撤曲线使用组合持仓的不利价格估算：多头看当前 K 的最低价，空头看当前 K 的最高价；退出 K 会先按仍持仓状态估算，再按结算净值确认修复。

本地运行

首次运行先安装依赖，再启动 Streamlit Web 应用。

cd /Users/a1234/Documents/TrendingWinning
python -m pip install -r requirements.txt
streamlit run streamlit_app.py --server.port 8520
Web 页面包含三个主栏目：TDX K线、策略扫描、回测。侧边栏只放全局数据目录、复权方式和 TDX 插件目录。Mac 本机不作为真实 TDX 取数环境，真实请求用 Parallels/Windows。

回测页的模块化字段和 K 线触发范例另见 backtest_kline_guide.html。

路径选择

所有文件夹路径都通过“选择文件夹”按钮弹出系统选择框，不再要求用户手写目录字符串。

位置用途交互方式
行情根目录本地 K 线仓库，例如 market/daily。点击“选择文件夹”弹出系统目录框；也可用快速位置、子文件夹和选定当前文件夹浏览。
TDX PYPlugins/userWindows 通达信 tqcenter 所在目录。Win 侧页面直接弹出本机目录框；Mac 侧 CLI 用 --runtime parallels 调度到 Win。
输出父目录保存回测或参数遍历产物。选择父目录，系统自动按实验名生成子文件夹。
如果当前会话没有可用桌面窗口或缺少 tkinter，页面会明确提示无法打开系统选择框，并继续提供站内目录浏览器。

TDX K线

先查看本地缓存库存，再用诊断确认 Windows/Parallels TDX 通道；随后做补齐计划，最后按需抓取缺失或质量异常的周期。

Mac 端 TDX 接口测试以 Parallels/Windows 通达信为准，先跑 tdx-doctor --runtime parallels，不要尝试从 Mac 本机通达信直接取数。若日 K 返回 ok、分钟 no_data，说明 Windows 通达信本地没有返回分钟 K 线；先确认 5m 分钟数据缓存，再做高周期请求或 5m 聚合。
python -m trending_winning.cli inventory-data \
  --symbols 000001.SZ,600519.SH \
  --timeframes 1d,5m,15m,30m,60m \
  --data-root /path/to/market/daily
python -m trending_winning.cli tdx-doctor \
  --symbols 000001.SZ,600519.SH \
  --timeframes 1d,5m,15m,30m,60m \
  --start "2026-05-25 09:30:00" \
  --end "2026-05-25 15:00:00" \
  --runtime parallels \
  --tdx-path "C:\\new_tdx64\\PYPlugins\\user"
python -m trending_winning.cli plan-data \
  --symbols 000001.SZ,600519.SH \
  --timeframes 1d,5m,15m,30m,60m \
  --start "2026-05-01 09:30:00" \
  --end "2026-05-25 15:00:00" \
  --min-coverage-ratio 0.95 \
  --data-root /path/to/market/daily
python -m trending_winning.cli prepare-data \
  --symbols 000001.SZ,600519.SH \
  --timeframes 1d,5m,15m,30m,60m \
  --start "2026-05-01 09:30:00" \
  --end "2026-05-25 15:00:00" \
  --min-coverage-ratio 0.95 \
  --data-root /path/to/market/daily
  • inventory-data 只列本地缓存库存；指定代码时缺失周期会显示 missing_file。
  • plan-data 只读取本地 parquet 和日 K 交易日锚点，不请求 TDX。
  • prepare-data 在 Mac 侧默认通过 Parallels 进入 Windows 执行真实 TDX 请求。
  • prepare-data 只请求缺失、质量错误或覆盖率低于门槛的标的周期。
  • 15m / 30m / 60m 原生周期无数据时，会按上午、下午交易段用 TDX 5m 聚合成目标周期。
  • 分钟周期会自动带上 1d 依赖，日 K 用于覆盖率锚点和涨停开盘过滤。
  • 所有入口会校验 start <= end，时间窗口写反会直接失败。

策略扫描

扫描页用于快速查看标志 K、趋势通道和突破触发。周期、标的、开始日期、结束日期在同一行，减少表单高度。

  • 周期：仅支持分钟级 5m / 15m / 30m / 60m。
  • 代码：支持逗号或换行分隔，例如 000001.SZ,600519.SH。
  • 结果：展示最新状态、触发数量和最近突破明细。

单策略回测

单策略只绑定一种形态识别模块，不进入组合仓位分配层。适合独立验证趋势、区间、通道或反转模块。

Web 参数区会显示“策略执行空间”：样本说明、识别形态、信号 K 条件、触发成交分类、大周期方向过滤、末端假突破过滤、结构止损、盈利通道回撤止盈、满仓进出、或然分支和复盘输出会集中列出，避免把组合仓位参数误认为单策略参数。

python -m trending_winning.cli single-backtest \
  --symbols 000001.SZ,600519.SH \
  --timeframe 30m \
  --higher-timeframe 60m \
  --higher-timeframe-max-age-minutes 120 \
  --start 2026-05-01 \
  --end 2026-05-25 \
  --detector trend \
  --risk-reward 2.0 \
  --max-holding-bars 12 \
  --max-actual-risk-pct 0.03 \
  --max-chase-pct 0.02 \
  --side-mode both \
  --min-coverage-ratio 0.95 \
  --fee-rate 0.0003 \
  --slippage-bps 5 \
  --enable-terminal-false-breakout-filter \
  --terminal-false-breakout-detectors trend,channel \
  --terminal-false-breakout-min-regime-bars 18 \
  --terminal-false-breakout-extension-atr-multiple 2.0 \
  --terminal-false-breakout-min-score 3 \
  --output-dir runs/single-trend-001

推荐验证顺序

  1. 先关闭大周期方向过滤，只验证形态本身。
  2. 再开启 higher_timeframe，检查方向不一致和大周期信号过旧的过滤日志。
  3. 需要处理趋势/通道末端假突破时，再开启末端假突破过滤；它只过滤开仓，不改变形态识别和撮合。
  4. 用 side_mode 在多/空、仅多、仅空之间切换，单独观察方向贡献。
  5. 最后加入真实风险、追价、手续费和滑点参数。

末端假突破过滤公式

过滤器默认关闭。启用后只作用于用户选择的模块，默认 trend,channel。多头信号用上轨、创新高推进和上影线；空头信号对称使用下轨、创新低推进和下影线。

terminal_score =
  1[regime_run >= min_regime_bars]
  + 1[abs(close - channel_mid) / ATR >= extension_atr_multiple]
  + 1[edge_count >= edge_min_count]
  + 1[progress_atr <= weak_progress_atr]
  + 1[wick_ratio >= wick_ratio]

if terminal_score >= min_score:
  reject reason = terminal_false_breakout_risk

其中 edge_count 表示最近窗口内多次贴近通道边缘，progress_atr 表示突破推进不足的程度，wick_ratio 表示上影线或下影线占整根 K 线振幅的比例。过滤只使用信号 K 及之前已完成 K 线，不使用未来走势。

组合策略回测

组合回测把多个独立形态生成的订单统一进入组合层，组合层只负责排序、容量、资金、行业和重叠持仓约束。

组合模式的“策略执行空间”会额外列出最大持仓、固定单笔仓位、风险预算、预留现金、空头保证金、策略优先级、策略资金上限、行业资金上限和同票重叠规则，用来确认组合层只处理资金分配，不重新识别形态。

python -m trending_winning.cli portfolio-backtest \
  --symbols 000001.SZ,600519.SH \
  --timeframe 30m \
  --start 2026-05-01 \
  --end 2026-05-25 \
  --detectors trend,range,channel,reversal \
  --risk-reward 2.0 \
  --max-open-positions 5 \
  --side-mode both \
  --capital-per-trade 0.25 \
  --risk-per-trade 0.01 \
  --reserve-cash 0.1 \
  --strategy-priority trend_signal_bar=1,range_signal_bar=2 \
  --strategy-capital-limit trend_signal_bar=0.6,range_signal_bar=0.4 \
  --sector-capital-limit 银行=0.5,新能源=0.4 \
  --symbol-sector-map 000001.SZ=银行,300750.SZ=新能源 \
  --output-dir runs/case-001 \
  --benchmark
  • capital_per_trade 是固定名义仓位。
  • risk_per_trade 按真实入场风险反推仓位。
  • allow_same_symbol_overlap 控制同一股票能否多策略重叠持仓。
  • side_mode 控制所有启用形态的交易方向：多/空、仅多或仅空。
  • 单策略和组合策略用信号 K 结构止损价与 risk_reward 计算目标平仓价；固定百分比止盈止损只属于旧突破回测。页面会显示“结构止损价说明”，可调止损参数是“结构止损最大风险”，用于限制开仓价到结构止损价的距离。
  • 盈亏比指向平仓信号，不决定开仓信号。开仓信号先给出开仓价和结构止损价；盈亏比只把这段风险距离换算成固定目标平仓价：多头目标平仓价 = 开仓价 + (开仓价 - 止损价) × 盈亏比，空头方向相反。
  • 盈利通道回撤止盈由页面开关显式启用；关闭时启动浮盈、最大盈利回撤幅度和当前周期均线周期都强制按 0 处理。启用时最大盈利回撤幅度可作为比例止盈单独使用，当前周期均线周期可作为均线回撤止盈单独使用，二者也可叠加；启动浮盈只是可选门槛，设为 0 表示不设门槛。最大盈利回撤幅度按上一根已完成 K 的最大盈利价位计算：例如多头入场 100、最高浮盈到 108，参数 0.020 时回撤线约 105.84，跌破即平仓；当前 K 只刷新下一根可用的回撤线，避免同一根 K 同时启动和退出。
  • 末端假突破过滤在组合中仍属于策略层过滤，先过滤开仓订单，再把剩余候选单送入组合仓位分配。组合层不重新识别趋势、通道、区间或反转。
  • 单策略按满仓进出，统计 market_bar_count 和 exposure_bar_ratio，用于判断策略在样本时间轴内实际占用资金多久。
  • 组合净值按逐 K 盯市，统计现金占比、净暴露、真实总暴露、持仓数和资金贡献。

勾选保存后，系统会把结果写到选定输出父目录下的实验子文件夹。

文件说明

trades.csv成交明细，含 order_id / event_id / event_type。
order_decisions.csv撮合层接受或拒绝原因；Web 结果页会把它汇总为“订单决策概览”和“拒绝原因分布”，先看信号是否触发入场价，再看最终成交和拒绝原因。
setup_order_decision_stats.csv按 detector_name / event_type / side 汇总撮合层接受、拒绝和止损风险指标；最终成交订单和触发成交候选订单在 stats.json 中分开汇总。
strategy_filter_decisions.csv策略层过滤原因，例如中部不交易、大周期方向不一致和末端假突破风险；末端过滤的拒绝原因为 terminal_false_breakout_risk。
setup_strategy_filter_stats.csv按信号形态汇总策略层过滤原因，区分信号 K 无流动性、大周期方向不一致、中部过滤和末端假突破过滤。
equity_curve.csv单策略和组合策略有成交时均输出逐 K 盯市净值；net_value 是正常净值，drawdown_net_value 是回撤估算净值，多头用 low、空头用 high，并按同一时刻所有开放持仓合成组合回撤；退出 K 会先按仍持仓状态估算不利价格，再按结算净值确认修复。
drawdown_curve.csv回撤曲线明细。path_net_value 是实际用于回撤计算的净值路径，point_type 区分 adverse_price（K 线不利价格估值）和 settlement（收盘或平仓后结算净值）；水下 K 数和修复 K 数按唯一 K 线时间点去重，平均回撤和 Ulcer 回撤压力指数按每根 K 的最差回撤计算。
drawdown_episodes.csv按回撤幅度从深到浅列出水下区间，含开始、触底、修复、回撤幅度、水下 K 数和是否修复；用于复盘每轮净值回撤。
strategy_stats.csv按 strategy_name 汇总策略成交绩效；已启用但没有成交的策略也保留零值行，便于单独评估策略是否无信号、全被拒或真实参与交易。
detector_stats.csv按 detector_name 汇总趋势、区间、通道、反转等识别模块的成交绩效；已启用但没有成交的模块也保留零值行，便于单独评估每个识别模块。
setup_stats.csv按 detector_name / event_type / side 汇总信号形态级别绩效，用于比较 H1/H2、失败突破、通道突破和反转信号；只有信号但全部被撮合拒绝或策略过滤的 setup 也保留零值行，同时带胜率区间、平均收益标准误和正期望概率，避免小样本误判。
signal_lifecycle_stats.csv按 detector_name / event_type / side / exit_reason 汇总开平仓路径绩效，用于观察同一类开仓信号最后主要通过目标平仓、盈利通道回撤止盈、结构止损还是持有到期退出。
trade_path_distribution.csv按持有 K 数、R 倍数、最大不利 R、最大有利 R 分桶汇总交易次数、胜率、平均收益和路径风险，用于判断收益主要来自哪个持仓周期和风险暴露区间。
symbol_stats.csv按股票汇总绩效，优先显示 stock_name，同时保留 stock_code 作为复核键；没有成交的样本股票也会保留零值行，便于区分“参与回测但无交易”和“未进入样本”。
data_inventory.csv本次实验涉及的日 K、主周期和高周期 parquet 缓存快照，包含存在状态、行数、起止时间、文件大小、缺失字段、修改时间和路径；库存扫描只读取 parquet 元数据和 date/stock_code 两列，不加载完整 OHLCV。
symbol_metadata.csv本次实验涉及股票的代码、名称和名称来源。优先读取行情目录 symbols.csv / stock_names.csv，其次读取 TDX hq_cache 的 shm/szm/bjm.tnf。
data_coverage.csv本地数据审计和覆盖率，包含 expected_rows、missing_rows、coverage_ratio、max_missing_gap_minutes、first_missing_at、last_missing_at、max_missing_gap_start_at 和 max_missing_gap_end_at，用于定位需要补齐的 K 线区间和最长连续缺口。关闭严格数据质量检查时，read_error 或 missing_columns 的 parquet 会保留审计记录并从回测数据包跳过，不会当成有效 K 线交易。
limit_filter_audit.csv日 K 涨停开盘过滤是否真实执行；daily_missing 表示日 K 缺失，daily_read_error / daily_missing_columns / daily_quality_error 表示日 K 文件损坏、缺字段或质量异常。
monthly_returns.csv自然周期收益、基于 drawdown_net_value 的周期最大回撤和净值样本数，是 stats.json 周期稳定性字段的明细来源。
summary.json参数遍历总览，包含 grid_case_count、case_count、grid_value_counts、pareto_case_count、best_case_name、best_case_config_hash、best_risk_adjusted_case_name、best_risk_adjusted_score、avg_risk_adjusted_score、median_risk_adjusted_score、worst_risk_adjusted_score、data_inventory_signature、data_inventory_unavailable_count、data_inventory_missing_columns_count、data_inventory_no_valid_rows_count、order_cache_hit_rate、candidate_cache_hit_rate、data_coverage_below_min_count、data_max_missing_gap_minutes、data_max_missing_gap_start_at、data_max_missing_gap_end_at、limit_filter_filtered_days、case_diagnostic_failed_count、case_diagnostic_attention_count、case_diagnostic_failed_case_count、case_strategy_row_count、case_strategy_trade_count、case_strategy_zero_trade_row_count、case_detector_row_count、case_detector_trade_count、case_detector_zero_trade_row_count、case_setup_row_count、case_setup_trade_count、case_setup_zero_trade_row_count、case_symbol_row_count、case_symbol_trade_count、case_symbol_zero_trade_row_count、case_setup_order_decision_count 和 case_setup_strategy_filter_decision_count；其中 grid_case_count 是原始组合数，case_count 是重复 grid 值展开前去重后的实际运行数。
pareto.csv从 sweep.csv 抽出的第一层非支配候选集，只包含 pareto_rank=1 的参数组。
parameter_summary.csv按每个 grid 参数值聚合结果，包含 case_count、pareto_case_count、pareto_hit_rate、positive_return_case_count、positive_return_rate、std_total_return、best_total_return、worst_total_return、best_sweep_rank、avg_risk_adjusted_score、median_risk_adjusted_score、avg_total_return、avg_max_drawdown、avg_monthly_worst_return、avg_monthly_return_std、avg_return_per_exposure_bar、avg_return_per_capital_exposure_bar、avg_return_per_margin_exposure_bar、avg_take_profit_exit_rate、avg_trailing_take_profit_exit_rate、avg_stop_loss_exit_rate、avg_max_holding_exit_rate、avg_acceptance_rate、avg_rejection_rate、avg_rejected_no_fill_count、avg_accepted_actual_risk_pct、avg_accepted_actual_chase_pct、avg_accepted_actual_reward_to_risk、avg_executed_actual_risk_pct、avg_executed_actual_chase_pct、avg_executed_actual_reward_to_risk、avg_strategy_filter_acceptance_rate 和 avg_strategy_filter_rejection_rate。风险质量评分用于比较参数区间的综合稳健性；参数遍历成交质量字段用于判断哪些参数更容易被撮合或策略过滤淘汰，避免只看平均收益选参数。
case_diagnostics.csv参数遍历每个 case 的实验诊断摘要，包含 sweep_rank、pareto_rank、case_config_hash、check、status、metric、value、threshold 和 detail，用于先筛掉数据覆盖差、无成交、回撤或路径风险过高的参数组。
case_strategy_stats.csv参数遍历每个 case 的策略级绩效，包含 sweep_rank / pareto_rank / case_config_hash / strategy_name；已启用但无成交的策略也保留零值行。
case_detector_stats.csv参数遍历每个 case 的识别模块级绩效，包含 sweep_rank / pareto_rank / case_config_hash / detector_name；已启用但无成交的趋势、区间、通道或反转模块也保留零值行。
case_setup_stats.csv参数遍历每个 case 的信号形态级绩效，包含 sweep_rank / pareto_rank / case_config_hash / detector_name / event_type / side；没有成交但出现过信号或拒单的 setup 也保留零值行。
case_symbol_stats.csv参数遍历每个 case 的股票级绩效，包含 sweep_rank / pareto_rank / case_config_hash / stock_name / stock_code；未成交样本股票也保留零值行，方便判断参数组在哪些股票上没有触发交易。
case_setup_order_decision_stats.csv参数遍历每个 case 的信号形态级撮合接受、拒绝和止损风险指标。
case_setup_strategy_filter_stats.csv参数遍历每个 case 的信号形态级策略过滤原因，保留过滤器和大周期方向维度。
sweep.csv参数遍历结果，首列 sweep_rank 按收益、回撤、月度稳定性、交易数和 case 名稳定排序；risk_adjusted_score 是 0-100 风险质量评分，用收益、组合回撤、月度最差收益、月度波动、Ulcer、单位持仓 K 效率和交易样本数的分位得分加权，再扣除诊断失败/关注惩罚；risk_adjusted_rank 是独立风险质量排名，不替换 sweep_rank。固定启用形态的 sweep 只保留启用形态、大周期 trend 上下文和通用撮合参数，未启用形态的参数不会生成额外 case，也不会改变 case_config_hash；case_config_hash 记录去除本机 name / data_root / output_dir 后的配置指纹，data_inventory_signature 记录本次 K 线缓存快照指纹；data_inventory_unavailable_count、data_inventory_missing_columns_count 和 data_inventory_no_valid_rows_count 用于判断缓存存在但不可用于回测的数量；pareto_rank=1 是收益、回撤、Ulcer、月度最差收益、月度收益波动和交易样本数的第一层非支配候选集，底层用批量支配矩阵分层；行内包含 diagnostic_failed_count、diagnostic_attention_count、diagnostic_primary_issue、monthly_win_rate、monthly_worst_return、monthly_worst_return_period、monthly_best_return_period、monthly_return_std、monthly_worst_drawdown_period、monthly_max_consecutive_losses、monthly_max_recovery_periods、monthly_current_underwater_periods 等诊断与周期稳定性字段。
case_configs.jsonl参数遍历每个 case 的完整配置，顺序与 sweep.csv 一致，可按 case_config_hash 回放。

需要复现某个参数组时，用 replay-case 读取 case_configs.jsonl，按 case_config_hash 重跑并保存新的回测产物，包括 equity_curve.csv、drawdown_curve.csv 和 drawdown_episodes.csv；若交易逻辑配置与哈希不一致会拒绝回放，本机路径和实验名不参与指纹。

python -m trending_winning.cli replay-case \
  --case-configs runs/sweep-001/case_configs.jsonl \
  --case-config-hash <case_config_hash> \
  --output-dir runs/replay-001

Docker

docker build -t trending-winning .
docker run --rm -p 8520:8501 \
  -v /Users/a1234/Desktop/trend-backtest/data:/data \
  trending-winning

容器内页面数据目录可选择 /data/market/daily。Docker 更适合读取已落地 parquet 后做扫描和回测，真实 TDX 取数仍走 Windows/Parallels。

Win/TDX 注意

  • 真实 TDX 请求依赖 Windows/Parallels 内通达信已启动并登录。
  • Mac CLI 默认 --runtime auto 会调度到 Parallels；需要本地强制运行时显式写 --runtime local。
  • 默认 Windows Python 是 C:\Users\Public\venvs\trending-winning\Scripts\python.exe，可用 --windows-python 覆盖。
  • 需要能定位 Windows PYPlugins/user，其中应能导入 tqcenter。
  • 在 Win 侧测试时，Web 文件夹选择器会浏览 Win 侧本机路径，不需要手写 macOS 路径。
  • 60m 会映射成 TDX 的 1h period。
  • 分钟线能否返回取决于客户端本地是否已有对应周期数据。
